# Code Style

## Загальні принципи

- використовувати серіалізовані приватні поля для налаштувань у `Inspector`
- перевіряти критичні залежності в `Awake()`
- вимикати компонент, якщо валідація конфігурації не пройдена
- віддавати перевагу коротким компонентам з однією відповідальністю

## Поточні патерни в коді

- `RequireComponent` для явних залежностей
- `Guard.Expect(...)` для runtime validation
- групування полів через `[Header(...)]`
- пояснення неочевидних serialized fields через `[Tooltip(...)]`
- числові обмеження в Inspector через `Min` або `Range`, якщо вони вже випливають з валідації або контракту поля
- зберігання налаштувань вводу через `InputActionReference`

## Оформлення Inspector

- упорядковувати serialized fields у тому порядку, в якому компонент реально налаштовують у Unity
- зверху розміщувати input та зовнішні references, нижче gameplay tuning, limits і safety-настройки
- використовувати `Header` для великих логічних блоків, а не для кожного одного поля
- додавати `Tooltip` лише для полів, значення яких неочевидне з назви
- використовувати `Min` або `Range` для числових полів, де некоректні значення вже забороняються кодом
- не перевантажувати прості компоненти зайвими `Header`, `Space` або декоративними атрибутами
- не змінювати runtime-логіку лише заради красивішого Inspector
- при перейменуванні serialized field зберігати існуючі дані prefab/scene через `FormerlySerializedAs`, якщо це потрібно

## Рекомендації для продовження

- не змішувати actor input, предметну логіку і stack assembly в одному класі
- нові gameplay systems додавати окремими компонентами
- зберігати однорідний стиль назв для prefab, material і script assets
- уникати прихованих runtime dependency, якщо їх можна виразити через `RequireComponent`
